# Contributing

This is not an attempt at a broad community policy document. It exists to pin down the release and changelog rules this
repository actually uses, so that humans and agents make the same choices.

## Commit Messages And PR Titles

Commit messages and PR titles must use Conventional Commit headers.

- Allowed types: `feat`, `fix`, `docs`, `doc`, `perf`, `refactor`, `style`, `test`, `chore`, `ci`, `revert`
- Scope is optional
- Type must reflect user-visible behavior, not implementation activity
- CLI interface or behavior changes (commands, flags or options, arguments, output contract, exit codes, documented
  usage) are never `refactor`; use `feat`, `fix`, or `perf` instead, and use `!` when the change is breaking

PR title enforcement and changelog decision validation live in `.github/workflows/conventional-commit-pr-title.yml`.

## Changelog Policy

Changelog generation uses git-cliff and the repository root `CHANGELOG.md`.

Every PR body must contain exactly one of these tags:

- `changelog: include`
- `changelog: skip`

The intent is simple: user-visible changes should usually be included, and non-user-visible work should usually be
skipped. Override tags win over commit type parsing. If both tags appear, `changelog: skip` wins.

## Releasing

Ask the user what version to bump to. Read the current version from `Cargo.toml`, then offer three options showing the
resulting version for each:

- Bump bugfix (e.g. 0.1.0 -> 0.1.1)
- Bump minor, reset bugfix (e.g. 0.1.1 -> 0.2.0)
- Bump major, reset minor+bugfix (e.g. 0.2.0 -> 1.0.0)

Then proceed:

1. Ensure you're on a fresh main with a clean working copy: `gt sync --all`, `gt checkout main`, then verify
   `git status` shows no uncommitted or untracked changes. Abort if dirty.
2. Set the version in `Cargo.toml`.
3. Refresh the lockfile: `cargo update --workspace`
4. Validate lockfile consistency: `cargo metadata --format-version 1 --locked > /dev/null`
5. Generate the changelog: `git-cliff --tag "v$VERSION" -o CHANGELOG.md`
6. Run `dprint fmt` to fix any formatting issues in the generated changelog.
7. Verify the changelog heading exists: `rg -n "^## \\[$VERSION\\]" CHANGELOG.md`
8. Create a release PR with commit message `chore: release $VERSION`. The PR must include `Cargo.toml`, `Cargo.lock`,
   and `CHANGELOG.md` (`CHANGELOG.md` will be untracked on first release; add it before committing).
9. Stop and explicitly ask the user for confirmation before merging and tagging. Do not silently wait. Say you are ready
   to merge and tag, and ask for confirmation.
10. Merge the PR: `gh pr merge <number> --squash`
11. Sync and checkout main: `gt sync --all`, `gt checkout main`.
12. Tag the merge commit and push: `git tag v$VERSION && git push origin v$VERSION`
13. Watch the Release workflow: `gh run watch <run-id>`. Confirm it succeeds (dist plan, release-plan tests, artifact
    builds, Homebrew formula publish).

## Preflight Checklist

1. Initialize git-cliff config if `cliff.toml` is missing:
   - `git cliff --init keepachangelog`
2. Ensure `dist-workspace.toml` has:
   - `tap = "scode/homebrew-dist-tap"`
   - `publish-jobs = ["homebrew"]`
   - `plan-jobs = ["./release-plan-tests"]`
3. Ensure repository secrets include:
   - `HOMEBREW_TAP_TOKEN`
4. Ensure `.github/workflows/release.yml` is generated by dist.
5. Ensure `.github/workflows/conventional-commit-pr-title.yml` exists.
6. Ensure `cliff.toml` exists and is configured for Conventional Commit parsing.
